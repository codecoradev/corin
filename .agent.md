## Project

Codecora Hub ÔÇö desktop knowledge workstation powered by codecora.dev.
Local-first, offline-capable Tauri 2 + Svelte 5 + Rust + SQLite app for managing
memories, knowledge graphs, and rooms.

## Stack

| Layer | Tech |
|-------|------|
| Desktop shell | Tauri 2 |
| Frontend | Svelte 5 (runes: `$state`, `$derived`, `$effect`, `$props`) |
| Backend | Rust (edition 2024) |
| Storage | SQLite via rusqlite 0.40 (bundled) |
| Graph | D3.js v7 (force-directed) |
| Markdown | CodeMirror 6 + marked + dompurify |
| Theme | Catppuccin Mocha (dark) |

## ÔŤö Critical Rules ÔÇö MUST FOLLOW (read carefully)

### Rule 1: NEVER push directly to `develop` or `main`

All changes go through a feature branch Ôćĺ PR Ôćĺ CI green Ôćĺ merge.

```bash
# Ôťů CORRECT
git checkout -b feat/05-dashboard
# ... work ...
git push origin feat/05-dashboard
gh pr create --base develop

# ÔŁî FORBIDDEN
git checkout develop
git commit -m "..."
git push origin develop   # NEVER DO THIS
```

**Branch naming:**
- Feature: `feat/<issue#>-<description>` (e.g. `feat/05-dashboard`)
- Fix: `fix/<description>`
- Chore: `chore/<description>`

### Rule 2: NEVER bypass cora review (`--no-verify` is FORBIDDEN)

The pre-commit hook runs `cora review --staged`. If cora finds issues:

```bash
# Ôťů CORRECT ÔÇö fix the issues, then commit again
# (review the cora output, fix the code, re-stage, commit)

# ÔŁî FORBIDDEN
git commit --no-verify   # NEVER DO THIS
```

If cora flags a false positive on generated files (e.g. `src-tauri/gen/`),
add them to `ignore.files` in `.cora.yaml` instead of bypassing.

### Rule 3: `cargo fmt` + `cargo clippy` before every commit

```bash
cd src-tauri && cargo fmt && cargo clippy --all-targets -- -D warnings
```

CI runs `--check` and `-D warnings` ÔÇö any formatting or lint failure blocks the PR.

### Rule 4: One issue = one branch = one PR = wait CI green = merge = next

Do not combine multiple issues in one branch. Do not start the next issue
until the current PR is merged and CI is green.

### Rule 5: PRs always target `develop` (never `main`)

`main` is only for releases. Release flow: develop Ôćĺ PR to main Ôćĺ tag Ôćĺ release.

## Workflow (per issue)

```
1. git checkout develop && git pull
2. git checkout -b feat/<issue#>-<description>
3. ... implement ...
4. cd src-tauri && cargo fmt && cargo clippy --all-targets -- -D warnings
5. npm run build   # verify frontend compiles
6. git add <files>
7. git commit -m "feat: ..."   # cora review runs automatically
8. git push origin feat/<branch>
9. gh pr create --base develop --title "..." --body "Closes #N"
10. wait for CI green (all checks pass)
11. merge PR (squash)
12. git checkout develop && git pull
13. delete local branch
14. proceed to next issue
```

## Key Rules for Agents

- **Svelte 5 runes** (`$state`, `$derived`, `$effect`) are correct ÔÇö do NOT use Svelte 4 `$:` syntax
- **All SQL** must use parameterized queries (`rusqlite::params![]` or `?N` placeholders)
- **rusqlite 0.40:** `usize` does NOT implement `ToSql`/`FromSql` ÔÇö always cast to `i64`
- **Tauri commands** are async fns with `#[tauri::command]`, state via `tauri::State<Arc<Mutex<AppState>>>`
- **`Manager` trait** must be imported (`use tauri::Manager`) to use `get_webview_window`
- **IPC types** must stay in sync between `src/lib/ts/types.ts` and `src-tauri/src/commands.rs`
- **CommandError** serializes to string for frontend ÔÇö add new variants carefully

## Code Review

- **Pre-commit hook:** `cora review --staged --format compact` runs automatically
- **CI:** `cora-review.yml` runs on every PR via `codecoradev/cora-review-action@v1`
- Config: `.cora.yaml` (provider, focus, rules, ignore patterns)
- Required CI secrets: `CORA_API_KEY`, `CORA_BASE_URL`, `CORA_MODEL`
