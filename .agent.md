### ŌÜĀ’ĖÅ CRITICAL: All Uteke Access via HTTP API Only

**NEVER open a direct SQLite connection to `~/.uteke/uteke.db`.**
All Uteke data (memories, graph, rooms, stats, search) must go through
the HTTP client (`uteke_client.rs` ŌåÆ `http://127.0.0.1:8767`).

Reasons:
1. **Always fresh** ŌĆö HTTP reads latest DB state, no stale cache
2. **Future-proof** ŌĆö server URL can be `localhost` or `https://uteke.yourdomain.com`
3. **Consistency** ŌĆö delete via HTTP + stats via HTTP = data always sync
4. **Remote support** ŌĆö desktop app can connect to remote uteke-serve

If tempted to add `rusqlite::Connection::open("~/.uteke/uteke.db")` ŌĆö **STOP**.
Use `UtekeClient` instead.

### ŌÜĀ’ĖÅ CRITICAL: Room/CRUD commands MUST use HTTP client (not native store)

The native store (`state.uteke` / `ensure_uteke()`) is **never initialized** ŌĆö
`s.uteke` is always `None`. Only `s.uteke_client` (the HTTP client) is set
in `setup()`. So:

- **WRONG:** `s.ensure_uteke()?.store().create_room(...)` ŌåÆ
  fails with `"uteke store not initialized"`
- **RIGHT:** clone the HTTP client out of the lock, then call the HTTP method:
  ```rust
  let client = {
      let s = state.lock().await;
      s.uteke_client.clone()
  }.ok_or(CommandError::NotInitialized)?;
  client.room_create(&id, Some(&name), ns).await.map_err(CommandError::Uteke)?;
  ```

Every room/CRUD command must follow this clone-client-out-of-lock pattern.

### Response decoding gotcha

`uteke-serve` returns JSON **objects** (e.g. `{"created":"...","namespace":"..."}`),
not plain strings. When adding client methods:
- Deserialize to `serde_json::Value` (safe, flexible), OR
- Define a typed struct matching the response
- **NEVER** deserialize to `String` ŌĆö it will fail with
  `"error decoding response body"`

### Connection layer (Phase 6+)

Remote/local connections live in the `connections` table and are managed via the
`connections/` module.  Rules:

1. **Active backend = `AppState.uteke_client`.**  It is rebuilt from the primary
   connection by `rebuild_active_client()` / the `reconnect_connection(id)`
   command.  Switching primary (`set_primary_connection`) **also rebuilds the
   live client** ŌĆö never update the DB row without swapping the client.
2. **Local vs remote** is decided by `config::is_remote_url()`.  Local URLs
   trigger `ensure_uteke_server()` (install + spawn); remote URLs skip spawn.
3. **Auth tokens are secret.**  Never log a token verbatim ŌĆö use
   `mask_token_log()`.  On `delete_connection`, `store::clear_token()` wipes the
   column to NULL **before** the row is deleted (prevents forensic recovery).
4. **DB file perms** are set to `0600` on startup (Unix) because the db holds
   tokens.
5. **Adding a product** = new `ProductType` variant + adapter impl + register in
   `build_adapter()`.  Do not touch connection CRUD/persistence/UI plumbing.
6. **Disconnect vs reconnect.** `disconnect_connection` drops
   `AppState.uteke_client` to `None` (recall/search fail until reconnect) but
   keeps the primary row + flag.  `reconnect_connection(id)` rebuilds it live.
7. **Set Primary guard.** The UI only offers "Set Primary" on connections whose
   status is `connected`, and "Disconnect" only on the connected primary.
8. **No native `confirm()`/`alert()`** ŌĆö Tauri's webview blocks them. Use an
   in-app modal (see the `confirm-overlay` / `confirm-dialog` pattern in
   `ConnectionManager.svelte`).
9. **Network is the bottleneck for remote backends.** Never fan out one HTTP
   call per namespace when a cross-namespace call exists: `/recall` is
   cross-namespace (uteke #448 fixed) ŌĆö use a single call. `/list` is **not**
   cross-namespace yet (uteke #526) ŌĆö page within a namespace.
10. **Cache reusable reads.** Small, frequently-reused responses (namespaces,
    stats) go through `stores/cache.svelte.ts` with a short TTL and are
    invalidated on mutations (remember/forget/reconnect/disconnect/set-primary).
    Do not add a DB/Redis layer ŌĆö this is a desktop app, in-process only.
11. **Paginate, don't over-fetch.** List views use `stores/pagination.svelte.ts`
    with a 20-item page size and a "Load more" button. Never fetch
    `limit: 500` just to count.

### Error handling pattern (frontend CRUD)

**NEVER** silently swallow errors with `catch { }`. Every CRUD operation must:
1. `catch (e)` and extract the message
2. `console.error(op + ' failed', e)`
3. Surface to the user via an error banner/state (e.g. `reportError()` helper)

Silent catches hide real bugs (wrong endpoint, type mismatch, server down)
and make debugging impossible.

## Ōøö Critical Rules ŌĆö MUST FOLLOW (read carefully)

### Rule 1: NEVER push directly to `develop` or `main`

All changes go through a feature branch ŌåÆ PR ŌåÆ CI green ŌåÆ merge.

```bash
# Ō£ģ CORRECT
git checkout -b feat/05-dashboard
# ... work ...
git push origin feat/05-dashboard
gh pr create --base develop

# ŌØī FORBIDDEN
git checkout develop
git commit -m "..."
git push origin develop   # NEVER DO THIS
```

**Branch naming:**
- Feature: `feat/<issue#>-<description>` (e.g. `feat/05-dashboard`)
- Fix: `fix/<description>`
- Chore: `chore/<description>`

### Rule 2: NEVER bypass cora review (`--no-verify` is FORBIDDEN)

The pre-commit hook runs `cora review --staged`. If cora finds issues:

```bash
# Ō£ģ CORRECT ŌĆö fix the issues, then commit again
# (review the cora output, fix the code, re-stage, commit)

# ŌØī FORBIDDEN
git commit --no-verify   # NEVER DO THIS
```

If cora flags a false positive on generated files (e.g. `src-tauri/gen/`),
add them to `ignore.files` in `.cora.yaml` instead of bypassing.

### Rule 3: `cargo fmt` + `cargo clippy` before every commit

```bash
cd src-tauri && cargo fmt && cargo clippy --all-targets -- -D warnings
```

CI runs `--check` and `-D warnings` ŌĆö any formatting or lint failure blocks the PR.

### Rule 4: One issue = one branch = one PR = wait CI green = merge = next

Do not combine multiple issues in one branch. Do not start the next issue
until the current PR is merged and CI is green.

### Rule 5: PRs always target `develop` (never `main`)

`main` is only for releases. Release flow: develop ŌåÆ PR to main ŌåÆ tag ŌåÆ release.

## Key Rules for Agents

- **Svelte 5 runes** (`$state`, `$derived`, `$effect`) are correct ŌĆö do NOT use Svelte 4 `$:` syntax
- **All SQL** must use parameterized queries (`rusqlite::params![]` or `?N` placeholders)
- **rusqlite 0.40:** `usize` does NOT implement `ToSql`/`FromSql` ŌĆö always cast to `i64`
- **Tauri commands** are async fns with `#[tauri::command]`, state via `tauri::State<Arc<Mutex<AppState>>>`
- **Tauri arg naming:** Rust `snake_case` params are auto-converted to `camelCase` at the
  IPC boundary. Rust `room_id: String` ŌåÆ frontend must invoke with `{ roomId }`, NOT `{ room_id }`.
  Single-word params (`name`, `tags`) are unaffected. Mismatched keys produce:
  `invalid args ... missing required key roomId`
- **`Manager` trait** must be imported (`use tauri::Manager`) to use `get_webview_window`
- **IPC types** must stay in sync between `src/lib/ts/types.ts` and `src-tauri/src/commands.rs`
- **CommandError** serializes to string for frontend ŌĆö add new variants carefully
